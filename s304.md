# 🔒 AWS S3 — Security & Access Control


---

## 📌 Why Security Matters in S3?

- S3 buckets are **private by default**
- Misconfigured buckets = AWS data breach
- You control **who can access what** and **how**

### 3 Ways to Control Access
```
1. Bucket Policy   →  JSON rules on the bucket (resource-based)
2. IAM Policy      →  Rules attached to users/roles (identity-based)
3. Block Public Access → Master override switch
```

---

## 🛡️ Part 1 — Block Public Access

The **master safety switch** for your bucket.

```
S3 → Bucket → Permissions → Block Public Access
```

### 4 Settings Explained

| Setting | What It Does |
|---------|-------------|
| `BlockPublicAcls` | Blocks NEW public ACLs being added |
| `IgnorePublicAcls` | Ignores ALL existing public ACLs |
| `BlockPublicPolicy` | Blocks NEW public bucket policies |
| `RestrictPublicBuckets` | Restricts access via any public policy |

> ✅ Keep ALL FOUR ON unless hosting a public static website

## 🪣 Part 2 — Bucket Policy

JSON-based rules that control access **to the bucket**.

```
S3 → Bucket → Permissions → Bucket Policy
```

### Policy Structure
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect":    "Allow or Deny",
      "Principal": "who",
      "Action":    "what",
      "Resource":  "which bucket/object"
    }
  ]
}
```

### Example 1 — Make Bucket Public (Read Only)
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```
> ⚠️ `"Principal": "*"` = everyone in the world. Use carefully!

### Example 2 — Allow Specific IAM User Only
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:user/john"
      },
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

### Example 3 — Deny Delete for Everyone
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:DeleteObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

## 👤 Part 3 — IAM Policy vs Bucket Policy

| | IAM Policy | Bucket Policy |
|--|-----------|---------------|
| Attached to | User / Role / Group | Bucket |
| Controls | What the user can do | Who can access bucket |
| Cross-account? | ❌ Same account only | ✅ Yes |
| Use when | Managing your own users | Granting access to others |

### When to Use Which?
```
Your own IAM users need S3 access  →  Use IAM Policy
Public access needed               →  Use Bucket Policy
Cross-account access               →  Use Bucket Policy
Restrict specific bucket actions   →  Use Bucket Policy
```

## 🔗 Part 4 — Pre-Signed URLs

Give **temporary access** to a private object — without making it public.

### How It Works
```
User requests file
      ↓
Your app generates Pre-Signed URL (expires in X seconds)
      ↓
User downloads file directly from S3
      ↓
URL expires → access gone
```

### Generate via CLI
```bash
# Generate URL valid for 1 hour (3600 seconds)
aws s3 presign s3://my-bucket/private-file.pdf --expires-in 3600

# Generate URL valid for 5 minutes
aws s3 presign s3://my-bucket/private-file.pdf --expires-in 300
```

### Output
```
https://my-bucket.s3.ap-south-1.amazonaws.com/private-file.pdf
?X-Amz-Algorithm=AWS4-HMAC-SHA256
&X-Amz-Expires=3600
&X-Amz-Signature=abc123...
```

> ✅ Share this URL — it works for 1 hour, then expires automatically

### Key Points
```
✅ Works on private objects (no public access needed)
✅ Expiry: seconds to 7 days max
✅ Use case: file downloads, secure uploads, sharing reports
❌ Don't use for permanent public access → use bucket policy instead
```

## 🔐 Part 5 — Encryption

Every object stored in S3 can be **encrypted at rest**.

### 4 Encryption Options

| Type | Key Managed By | Notes |
|------|---------------|-------|
| **SSE-S3** | AWS (automatic) | Default. Zero setup. |
| **SSE-KMS** | AWS KMS (you control) | Audit trail via CloudTrail |
| **SSE-C** | You (customer) | You send key with every request |
| **Client-side** | You | Encrypt before uploading |

### Enable Default Encryption (Console)
```
S3 → Bucket → Properties → Default Encryption
→ Enable
→ Choose: SSE-S3 (recommended for most use cases)
→ Save
```

### Upload with Encryption via CLI
```bash
# SSE-S3 encryption
aws s3 cp file.txt s3://my-bucket/ --sse AES256

# SSE-KMS encryption
aws s3 cp file.txt s3://my-bucket/ --sse aws:kms
```

> ✅ Encryption in transit is automatic — S3 always uses HTTPS

## ❌ Common Mistakes

| Mistake | Risk | Fix |
|---------|------|-----|
| `"Principal": "*"` on sensitive data | Data breach | Use specific ARN |
| Turning off Block Public Access globally | Accidental exposure | Only disable when needed |
| No expiry on pre-signed URLs | Permanent leak | Always set short expiry |
| Storing secrets in S3 | Credentials exposed | Use AWS Secrets Manager |
| No encryption on sensitive data | Compliance failure | Enable SSE-S3 by default |

## 📝 Interview Questions

**Q: What is the difference between IAM Policy and Bucket Policy?**
> IAM Policy is attached to a user/role and controls what they can do. Bucket Policy is attached to the bucket and controls who can access it. Both must allow for access to be granted.

**Q: What is a Pre-Signed URL?**
> A temporary URL that grants time-limited access to a private S3 object without making the bucket public. Expires after a set time (max 7 days).

**Q: What happens if IAM Policy allows but Bucket Policy denies?**
> **Deny always wins.** An explicit Deny overrides any Allow in AWS.

**Q: What is SSE-S3?**
> Server-Side Encryption using S3-managed keys. AWS handles everything automatically. Default encryption for all new buckets.

**Q: How do you share a private S3 file temporarily?**
> Generate a Pre-Signed URL using AWS CLI or SDK with an expiry time.

## ⭐ Summary

```
Block Public Access  →  Master switch — keep ON by default
Bucket Policy        →  JSON rules — who can access your bucket
IAM Policy           →  Attach to users — what they can do in S3
Pre-Signed URL       →  Temporary access to private objects
Encryption           →  SSE-S3 (default) | SSE-KMS (audit trail)

Golden Rule: Deny always overrides Allow in AWS
```
