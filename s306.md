# S3 Versioning, Replication (CRR & SRR) and Lifecycle Rules

---

## 1. S3 Versioning

### What is Versioning?

Versioning is a feature in S3 that keeps **multiple versions of the same object** in a bucket. Every time you upload, overwrite, or delete an object, S3 stores a new version instead of permanently replacing or deleting it.

### Why Use Versioning?

- Protect against accidental deletion or overwrite
- Recover older versions of files at any time
- Required for enabling replication (CRR / SRR)
- Required for enabling MFA Delete

### Versioning States

A bucket can be in one of three states:

| State | Meaning |
|---|---|
| Unversioned (default) | No versioning, objects have no version ID |
| Versioning-enabled | All objects get a unique version ID |
| Versioning-suspended | New objects get null version ID, old versions preserved |

> **Note:** Once you enable versioning on a bucket, you can suspend it but you can **never fully disable** it back to the unversioned state.

### How Versioning Works

**Upload same key multiple times:**

```
PUT image.jpg  →  Version ID: abc123
PUT image.jpg  →  Version ID: def456  (latest)
PUT image.jpg  →  Version ID: ghi789  (latest)
```

All three versions are stored. The latest version is served by default when you GET the object.

**Delete an object with versioning enabled:**

When you delete without specifying a version ID, S3 does **not** permanently delete. It places a **Delete Marker** on top.

```
DELETE image.jpg  →  Adds Delete Marker (version ID: xyz000)
```

The object appears deleted but all older versions still exist. To restore, delete the Delete Marker.

**Permanent delete:**

To permanently delete a specific version, you must specify the version ID:

```
DELETE image.jpg?versionId=abc123  →  Permanently removes that version
```

### How to Enable Versioning (Console Steps)

1. Go to **S3 Console** → Open your bucket
2. Click **Properties** tab
3. Scroll to **Bucket Versioning**
4. Click **Edit** → Select **Enable**
5. Click **Save changes**

### MFA Delete

An extra layer of protection on top of versioning. When enabled, permanently deleting a version or changing the versioning state requires MFA authentication.

- Can only be enabled via AWS CLI (not console)
- Only the root account can enable MFA Delete

## 2. S3 Replication

### What is Replication?

S3 Replication automatically copies objects from a **source bucket** to a **destination bucket** based on replication rules. It requires versioning to be enabled on the source bucket.

### Types of Replication

| Type | Full Form | Region | Account |
|---|---|---|---|
| CRR | Cross-Region Replication | Different region | Same or different account |
| SRR | Same-Region Replication | Same region | Same or different account |

---

## 3. CRR — Cross-Region Replication

### What is CRR?

CRR copies objects from a source bucket in **one AWS region** to a destination bucket in a **completely different AWS region** automatically.

```
Source Bucket         →        Destination Bucket
us-east-1 (N.Virginia)         ap-south-1 (Mumbai)
```

### Use Cases

- **Disaster Recovery** — Keep a copy of data in another region in case one region goes down
- **Low Latency** — Serve users from a bucket closer to them geographically
- **Compliance** — Some regulations require data copies stored in specific regions
- **Data Sovereignty** — Keep data in a country-specific region as required by law

### Requirements for CRR

- Versioning must be **enabled** on both source and destination buckets
- An **IAM role** with permissions to read from source and write to destination
- Source and destination buckets must be in **different regions**
- Buckets can be in the **same or different AWS accounts**

### What Gets Replicated

- New objects uploaded after replication is configured
- Object metadata
- Object ACLs (if enabled)
- Object tags
- Delete markers (optional, must be explicitly enabled)

### What Does NOT Get Replicated

- Objects uploaded **before** replication was enabled
- Objects in Glacier or Glacier Deep Archive storage class (by default)
- Objects encrypted with SSE-C (customer-provided keys)
- Replicated objects from another replication rule (no chaining by default)

### How to Set Up CRR (Console Steps)

1. **Enable versioning** on source bucket
2. **Create destination bucket** in a different region and enable versioning on it
3. Go to source bucket → **Management** tab → **Replication rules**
4. Click **Create replication rule**
5. Set rule name, status = Enabled
6. Choose **source** (entire bucket or by prefix/tag)
7. Choose **destination bucket** (different region)
8. Choose or create an **IAM role**
9. Optionally enable **delete marker replication**
10. Save

## 4. SRR — Same-Region Replication

### What is SRR?

SRR copies objects from a source bucket to a destination bucket within the **same AWS region**.

```
Source Bucket         →        Destination Bucket
us-east-1             →        us-east-1
(prod-bucket)                  (logs-archive-bucket)
```

### Use Cases

- **Log aggregation** — Copy logs from multiple buckets into a single centralized bucket
- **Test and production separation** — Replicate prod data into a test bucket in the same region
- **Cross-account copy** — Copy data from one AWS account to another account in the same region
- **Compliance copy** — Maintain a second copy within the same region for audit purposes

### Requirements for SRR

- Versioning must be **enabled** on both source and destination buckets
- An **IAM role** with permissions to read from source and write to destination
- Source and destination buckets must be in the **same region**

### How to Set Up SRR (Console Steps)

Same steps as CRR but:
- Destination bucket must be in the **same region** as source
- In Step 7, select a bucket in the **same region**

---

## 5. CRR vs SRR — Quick Comparison

| Feature | CRR | SRR |
|---|---|---|
| Region | Different region | Same region |
| Account | Same or different | Same or different |
| Versioning required | Yes | Yes |
| Disaster recovery | Yes (primary use case) | No |
| Log aggregation | No | Yes (primary use case) |
| Latency benefit | Yes (geo-distributed users) | No |
| Compliance (geo) | Yes | Limited |
| Cost | Higher (cross-region data transfer fees) | Lower (no cross-region transfer fee) |

## 7. S3 Lifecycle Rules

### What are Lifecycle Rules?

Lifecycle rules allow you to **automatically manage objects** in your S3 bucket over time — either by transitioning them to a cheaper storage class or by expiring (deleting) them after a defined number of days.

This helps **reduce storage costs** without manually managing objects.

### Two Types of Lifecycle Actions

| Action | What it Does |
|---|---|
| **Transition** | Move objects to a cheaper storage class after N days |
| **Expiration** | Permanently delete objects after N days |

### S3 Storage Classes (from expensive to cheapest)

```
S3 Standard
    ↓ (after 30 days)
S3 Standard-IA (Infrequent Access)
    ↓ (after 60 days)
S3 Intelligent-Tiering
    ↓ (after 90 days)
S3 One Zone-IA
    ↓ (after 180 days)
S3 Glacier Instant Retrieval
    ↓ (after 365 days)
S3 Glacier Flexible Retrieval
    ↓ (after 730 days)
S3 Glacier Deep Archive  ← cheapest, slowest retrieval
```

> You can only transition objects **downward** (to cheaper classes). You cannot move from Glacier back to Standard using lifecycle — you would need to restore first.

### Common Lifecycle Rule Examples

**Example 1 — Archive old logs automatically:**

```
Day 0    → Object uploaded to S3 Standard
Day 30   → Move to S3 Standard-IA
Day 90   → Move to S3 Glacier
Day 365  → Delete permanently
```

**Example 2 — Delete incomplete multipart uploads:**

```
After 7 days → Delete incomplete multipart uploads automatically
```

This is a common and recommended lifecycle rule to avoid paying for abandoned multipart upload parts.

**Example 3 — Clean up old object versions (with versioning enabled):**

```
Non-current versions older than 30 days → Delete permanently
```

This is important when versioning is enabled — without this, old versions accumulate and increase storage costs.

### Lifecycle Rule Scope

You can apply a lifecycle rule to:

| Scope | Description |
|---|---|
| Entire bucket | Rule applies to all objects |
| By prefix | e.g., only objects under `logs/` |
| By object tag | e.g., only objects tagged `env=dev` |

### How to Create a Lifecycle Rule (Console Steps)

1. Go to **S3 Console** → Open your bucket
2. Click **Management** tab
3. Click **Create lifecycle rule**
4. Enter a **rule name**
5. Choose **scope** (entire bucket or filter by prefix/tag)
6. Select actions:
   - Transition current versions
   - Transition non-current versions (if versioning enabled)
   - Expire current versions
   - Delete non-current versions
   - Delete expired object delete markers
   - Delete incomplete multipart uploads
7. Set the **number of days** for each action
8. Review and **Create rule**

### Lifecycle Rules with Versioning

When versioning is enabled, lifecycle rules can manage both current and non-current (older) versions separately:

| Action | Applies To |
|---|---|
| Transition current versions | Latest version of each object |
| Transition non-current versions | All older/previous versions |
| Expire current versions | Adds a delete marker on current version |
| Delete non-current versions | Permanently removes older versions |
| Delete expired delete markers | Cleans up delete markers with no versions left |

## 8. Key Points to Remember for Teaching

- Versioning keeps all object versions — a delete only adds a delete marker, not a true delete
- Versioning is **mandatory** before enabling any replication (CRR or SRR)
- CRR = different region, SRR = same region — both can be same or different account
- Cross-account replication needs a bucket policy on the destination bucket
- Replication only copies **new** objects — existing objects are not replicated automatically
- Lifecycle rules save cost by moving objects to cheaper storage over time
- Always add a lifecycle rule to delete **incomplete multipart uploads** (cost saving tip)
- Always add a lifecycle rule to delete **non-current versions** when versioning is enabled (cost saving tip)
- Lifecycle transitions go **only downward** — Standard → IA → Glacier, never the reverse
